# Week 2 과제: 택시 호출 서비스 설계

## 1. 문제 이해 및 설계 범위 확정

### 시나리오

승객이 호출하면 주변 빈 택시를 찾아 매칭하고, 매칭된 택시가 픽업 지점에 도착할 때까지 승객은 택시 위치와 도착 예상 시간을 실시간으로 확인한다. 픽업 이후 목적지에 도착할 때까지 위치 추적이 이어진다.


### 설계 범위 (In / Out of Scope)

| 포함 (In Scope) | 제외 (Out of Scope) |
|---|---|
| 호출 시점부터 도착 완료 시점까지의 실시간 흐름 | 회원가입 · 인증 · 기사 등록 절차 |
| 기사·승객 위치 추적 | 이용 이력 |
| 매칭 로직 | 운행 종료 후 요금 산정 · 결제 · 리뷰 · 정산 |
| 이상 상황 처리 (앱 종료, 네트워크 단절, 취소 등) | 도로망 데이터 / 경로 탐색 알고리즘 |

### 시스템 구성 전제
- 우리가 설계할 오로지 택시 호출 서비스(OOO택시)이다.
- 외부 의존: 지도 시스템(OOO맵, 경로·ETA), 회원 시스템, 결제 시스템
- 기사·승객 모두 로그인 상태, 결제 수단 등록 완료 상태로 가정
- 외부 시스템 의존에서 오는 문제는 본 서비스가 책임진다

### 기능 요구사항
- 기사 위치를 서버에 저장하고, 승객 호출 시 반경 내 빈 택시를 가까운 순으로 매칭
- 매칭 후 픽업 이동 중: 승객 화면에 택시 위치 + 픽업 지점까지 경로·ETA 실시간 표시
- 운행 중: 승객 화면에 현재 위치 + 목적지까지 경로·ETA 실시간 표시
- 이상 상황(앱 종료·네트워크 단절·호출 취소)에서 시스템이 일관된 상태 유지

### 비기능 요구사항 (시간 / 지연 목표)

| 항목 | 목표 |
|---|---|
| 호출 접수 응답 시간 | 호출 요청 → "기사 검색 중" 진입까지 **2초 이내** |
| 매칭 완료 시간 | 평균 **30초 이내**, 5분 초과 시 실패 처리 |
| 위치 추적 갱신 지연 | 기사 위치 변화 → 승객 화면 반영 평균 **5초 이내** |

### 개략적 규모 추정 _(기준값 — 본인 가정으로 변경 가능)_

| 항목 | 수치 |
|---|---|
| 서비스 지역 | 단일 대도시권 |
| 누적 가입 승객 | 약 2,000,000명 |
| MAU / DAU | 약 800,000명 / 약 200,000명 |
| 누적 가입 기사 | 약 50,000명 |
| 동시 운행 기사 | 10,000명 |
| 일일 호출 수 | 약 500,000건 |
| 피크 시간 호출 집중도 | 평균 대비 **5배 이상** |
| 피크 시간대 | 평일 출근 07:30–09:30 / 퇴근 18:00–20:00 / 금·토 심야 23:00–02:00 |

## 2. 개략적 설계안 제시 및 동의 구하기

### 핵심 흐름 (필수)

```
① 기사 앱이 주기적으로 위치 전송
② 승객이 호출 요청
③ 서버가 주변 빈 택시 검색 후 매칭 제안
④ 기사 수락 → 매칭 확정
⑤ 픽업 이동 중: 기사 위치를 승객 화면에 실시간 표시
⑥ 운행 중: 동일하게 실시간 위치 추적
⑦ 도착 완료 → 상태 종료
```

### 개략적 아키텍처 다이어그램 (필수)

```
기사 앱 ──HTTP/gRPC──► Location Service ──► Redis (Geo)
                              │
                              └──► Kafka ──► Location Processor ──► Redis
                                                      │
승객 앱 ◄──WebSocket── WebSocket Server ◄────────────┘

승객 앱 ──► API Gateway ──► Call Service ──► Trip DB (MySQL)
                              │
                              └──► Matching Service ──► Redis GEO / OOO맵
                              │
                              └──► Notification Service ──► 기사 앱 (푸시)
```

| 컴포넌트 | 역할 |
|---|---|
| API Gateway | 클라이언트 요청 인증·라우팅 |
| Call Service | 호출 생성·상태 관리 |
| Matching Service | 주변 기사 검색·매칭 처리 |
| Location Service | 기사 위치 수신·저장·조회 |
| Notification Service | 기사에게 매칭 제안 푸시 |
| WebSocket Server | 승객에게 실시간 위치 푸시 |
| Redis (Geo) | 기사 실시간 위치 저장 |
| Kafka | 위치 이벤트 비동기 버퍼링 |
| Trip DB (MySQL) | 호출 상태·운행 이력 저장 |
| OOO맵 (외부) | 경로 탐색, ETA 계산 |
| 회원 시스템 (외부) | 인증 |
| 결제 시스템 (외부) | 결제 |

## 3. 상세 설계

### 설계 대상 컴포넌트 사이의 우선순위 정하기 / 아키텍처 다이어그램 (필수)

**우선순위 (트래픽·일관성 관점)**

1. **Location Service + Redis GEO** — 위치 쓰기/반경 검색이 최대 부하 축이므로 최우선 튜닝·샤딩·복제 전략 대상.
2. **Call Service + Trip DB** — 호출 상태의 단일 진실 소스; 매칭·취소·완료의 원자적 전환.
3. **Matching Service + OOO맵** — ETA 비용·지연 관리(캐시·서킷브레이커·후보 제한).
4. **Kafka + Location Processor** — 위치 이벤트 스파이크 흡수, 재처리.
5. **WebSocket Server** — 승객 실시간 푸시; 스케일 아웃 시 연결 라우팅 필요.

> 아래 8가지 질문은 답변 **선택**이지만, 본인이 짚고 넘어가는 부분을 1~2개 골라 깊이 있게 다루는 것을 권장.

### 3-1. 기사 위치 업데이트 주기

상태에 따라 업데이트 주기를 다르게 설정한다.

| 상태 | 주기 | 이유 |
|---|---|---|
| 대기 중 | 10초 | 매칭 미발생, 정확도 조금 낮아도 됨 |
| 호출 제안 수신 | 3초 | 매칭 직전, 빠른 위치 파악 필요 |
| 픽업 이동 중 | 2초 | 승객이 화면에서 위치 확인 중 |
| 운행 중 | 2초 | ETA 정확도가 중요 |

**왜 상태별로 다른가?**

대기 중 기사 10,000명이 2초마다 위치를 올리면 초당 5,000건의 쓰기가 발생한다.
이 중 대부분은 매칭과 무관하다.
상태별 차등 주기로 비용과 정확도를 trade-off한다.

### 3-2. 기사 위치 저장 — 어디에, 어떤 형태로?

**선택: Redis + Geospatial Index**

```
# 기사 위치 저장
GEOADD driver:available 127.0275 37.4979 "driver_123"

# 반경 3km 내 기사를 거리순으로 조회
GEORADIUS driver:available 127.0275 37.4979 3 km ASC COUNT 10
```

**왜 Redis GEO인가?**

| 비교 | RDBMS | Redis GEO |
|---|---|---|
| 반경 검색 | 느림 (인덱스 있어도 복잡한 쿼리 필요) | 빠름 (네이티브 지원) |
| 쓰기 속도 | 상대적으로 느림 | 인메모리라 수 밀리초 |
| 영속성 | 강함 | 약함 (재시작 시 소실) |

위치 데이터는 현재값만 의미 있다. 5분 전 위치는 쓸모없다.
따라서 영속성보다 속도가 중요 → Redis가 적합하다.

Redis가 재시작되면? 기사 앱이 다음 주기에 다시 전송하면 된다.

**저장 구조**

```
Key: driver:available  → 대기 중인 기사 위치 집합
Key: driver:on_trip    → 운행 중인 기사 위치 집합
```

상태가 바뀌면 집합 간 이동 처리.

### 3-3. 검색 반경과 결과 처리

1. 기본 반경 1km로 검색
2. 결과 0건 → 3km → 5km → 10km 순으로 점진 확대
3. 10km에서도 없으면 "주변에 이용 가능한 택시가 없습니다" 안내

**왜 동적 반경인가?**

도심은 밀도가 높아 1km 안에 충분하지만, 외곽은 고정 반경이면 항상 실패한다.
불필요하게 넓은 반경으로 검색하면 API 비용이 늘어난다.
단계적 확장이 비용과 UX를 동시에 챙긴다.

### 3-4. 위치 · 경로 · ETA 전달 / 갱신 방식

**통신 방식**

| 구간 | 방식 | 이유 |
|---|---|---|
| 기사 앱 → Location Service | HTTP / gRPC | 단방향 업로드 중심, 모바일 환경에서 안정적 |
| WebSocket Server → 승객 앱 | WebSocket | 서버가 위치 변경 시 즉시 푸시 필요 |

**왜 승객 쪽에 WebSocket인가?**

HTTP polling은 승객이 "위치 바뀌었나요?"라고 주기적으로 서버에 물어봐야 한다.
승객 200,000명이 1초마다 물어보면 초당 200,000건의 불필요한 요청이 발생한다.
WebSocket은 연결을 유지한 채 변화가 생길 때만 서버가 푸시한다.

**위치 갱신 흐름**

```
기사 앱
  → Location Service
  → Kafka (버퍼링)
  → Location Processor
  → Redis 최신 위치 갱신
  → WebSocket Server
  → 승객 앱
```

**왜 중간에 Kafka가 있는가?**

위치 이벤트는 순간 폭주 가능성이 크다 (콘서트 종료, 폭우, 퇴근 시간).
Kafka로 버퍼링하면 순간 과부하를 흡수하고, 처리 실패 시 재처리가 가능하다.

### 3-5. 매칭 대상 선정 전략

**흐름**

```
① 주변 기사 후보 추출 (Redis GEO)
② 후보 기사에게 OOO맵으로 실제 ETA 계산
③ ETA 기준으로 정렬
④ 상위 5명에게 동시 제안
⑤ 가장 먼저 수락한 기사에게 Redis 분산락(SETNX)으로 매칭 확정
```

**왜 직선거리가 아닌 ETA인가?**

강 건너 기사, 유턴 불가 도로, 일방통행 때문에 직선거리가 가까워도 실제 도착이 늦을 수 있다.
Redis GEO로 1차 후보를 좁힌 뒤, OOO맵으로 실제 ETA를 계산하는 2단계 구조를 사용한다.

**왜 동시 제안 + 분산락인가?**

순차 제안(1명씩)은 안전하지만 느리다. 피크 타임에 매칭 시간이 길어진다.
동시 제안은 빠르지만 두 명이 동시에 수락하는 race condition이 발생한다.
`SETNX`(Set if Not Exists)로 첫 번째 수락자만 락을 획득하게 해서 중복 매칭을 방지한다.

```
SETNX trip:123 driver_456  # 성공 시 매칭 확정, 실패 시 이미 배정됨
```

### 3-6. 이상 상황 판단 · 처리

**상태(State) 관리**

택시 시스템은 상태 불일치가 가장 위험하다.

예시:
- 기사가 수락했는데 서버가 모름 → 다른 기사에게도 제안이 감 → 두 기사가 달려감
- 승객이 취소했는데 기사가 모름 → 기사는 계속 달려감

**상태 머신**

```
IDLE
  → MATCHING (호출 발생)
  → DRIVER_ASSIGNED (수락)
  → PICKUP (기사 이동 중)
  → RIDING (운행 중)
  → COMPLETED (도착)

예외:
  → CANCELLED (취소)
  → MATCH_FAILED (5분 내 매칭 실패)
  → DRIVER_TIMEOUT (기사 무응답)
```

**핵심 원칙**

상태 변경은 반드시 **원자적(Atomic)**이어야 한다.

`MATCHING → DRIVER_ASSIGNED` 전환 시 DB 트랜잭션 또는 Redis 분산락 사용.
두 요청이 동시에 같은 상태를 바꾸려 하면 하나만 성공해야 한다.

**기사 앱 종료 / 네트워크 단절**

| 상태 | 감지 기준 | 처리 |
|---|---|---|
| 대기 중 | 30초 이상 무보고 | Redis에서 제거, 오프라인 처리 |
| 픽업 이동 중 | 20초 이상 무보고 | 승객에게 "연결 불안정" 안내, 재연결 대기, 일정 시간 후 재배차 |
| 운행 중 | 60초 이상 무보고 | GPS 수신 문제 가능성 → 마지막 위치 유지, 재연결 허용, 이후 고객센터 fallback |

운행 중에 즉시 종료시키지 않는 이유: 터널, 지하주차장 등 GPS 수신이 잠깐 끊기는 경우가 흔하다.

**매칭 제안 무응답**

15초 내 응답 없으면 다음 후보 기사에게 제안.

**승객 호출 취소**

호출 상태를 CANCELLED로 변경 → 매칭 중인 기사에게 취소 알림 → 기사 상태 대기로 복구.

**승객 앱 종료**

호출 자체는 유지한다. 앱을 백그라운드로 내리는 경우가 흔하기 때문이다.

### 3-7. 외부 지도 / 경로 서비스 의존 관리

**문제**

OOO맵 API는 느릴 수 있고, 장애날 수 있고, 호출 비용이 비싸다.

**대응 전략**

**① 호출 최소화**

모든 기사에게 ETA를 계산하지 않는다.
Redis GEO로 1차 필터링 후 상위 후보만 ETA 계산.

**② 캐싱**

동일 출발지-목적지 경로는 Redis에 30초 캐시.
같은 경로를 여러 요청이 들어와도 OOO맵은 1번만 호출.

**③ 서킷브레이커 + Fallback**

OOO맵 장애 시:
- 서킷 오픈 → 호출 차단
- 직선거리 기반 임시 ETA 사용
- 승객에게 "도착 예정 시간 계산 중" 안내

정확도는 떨어지지만 서비스 자체는 유지된다.

### 3-8. 수요 폭주 예측 시 대응 _(예: 잠실 콘서트 종료 30분 내 2만 건 호출)_

**상황**: 잠실 아이유 콘서트 종료, 30분 내 2만 건 이상 호출 예상

이런 폭주는 행사 일정으로 사전 예측 가능하다.

**기술적 대응**

| 대응 | 내용 |
|---|---|
| 사전 Auto Scaling | 콘서트 종료 30분 전 Matching 서버·Kafka·WebSocket 서버 미리 증설 |
| Kafka 큐 버퍼링 | 호출 요청을 Kafka에 쌓고 순서대로 처리 → 스파이크 흡수 |
| 지역 캐시 워밍 | 잠실 주변 경로·ETA 결과를 미리 캐시 |
| 대기열 안내 | "현재 수요가 많아 대기 중, 약 N분 예상" 표시 |

**정책·운영적 대응**

| 대응 | 내용 |
|---|---|
| 기사 사전 유치 | 콘서트 종료 1시간 전 잠실 인근 기사에게 앱 알림 |
| 인센티브 제공 | 해당 시간대 잠실 운행 기사에게 추가 수수료 |
| Surge Pricing | 요금 올려 수요 분산, 기사 공급 유인 |

**트레이드오프**: 사전 스케일 업은 비용이 발생하지만, 대응 실패 시 CS 폭증과 브랜드 이미지 손상이 더 크다.

---

## 4. 설계 장점

- **Redis GEO**로 기사 위치 검색이 빠르고 대량 쓰기도 처리 가능
- **WebSocket**으로 서버→승객 실시간 푸시를 효율적으로 구현
- **Kafka**로 서비스 간 결합도를 낮추고 피크 트래픽을 버퍼링
- **Redis 분산락(SETNX)**으로 동시 수락 race condition 방지
- **서킷브레이커 + 캐싱**으로 외부 지도 API 장애를 흡수

---

## 5. 설계 단점

- **WebSocket 서버는 stateful**: 연결 상태를 유지해야 해서 스케일 아웃 시 세션 관리가 복잡해진다
- **Redis 소실 가능성**: 재시작 시 위치 데이터가 날아가지만, 기사 앱 재전송으로 복구된다 (순간 공백 존재)
- **ETA 정확도 한계**: OOO맵 장애 시 직선거리 기반으로 내려가 정확도가 떨어진다
- **상태 동기화 복잡성**: 분산 환경에서 race condition 관리는 항상 어렵다

---

## 6. 마무리


**① 매칭 속도 vs 안정성**

순차 제안은 안전하지만 느리다. 동시 제안 + 분산락으로 속도와 안정성을 동시에 챙겼다.

**② 실시간성 vs 서버 부하**

업데이트 주기를 짧게 할수록 정확하지만 서버 부하가 커진다. 상태별 차등 주기로 균형을 잡았다.

**③ 외부 의존 vs 자체 구현**

경로 탐색은 OOO맵에 위임하고 서킷브레이커·캐시로 장애를 흡수했다. 개발 비용은 낮지만 외부 장애에 종속된다.

모든 설계는 trade-off다. 정답은 없고, 요구사항과 제약 안에서 무엇을 포기하고 무엇을 지킬지 결정하는 것이 설계의 본질이다.


### 개인적 의견 / 사례 공유 / 추가 학습

이번 과제를 통해 택시 호출 서비스에서 가장 핵심적인 두 가지를 직접 느꼈다.

**첫째, 위치의 실시간성이 이 서비스의 본질이다.**

택시 호출 앱은 결국 "지금 기사가 어디 있는가"를 얼마나 빠르고 정확하게 전달하느냐가 전부다. 기사 위치가 늦게 반영되면 승객은 택시가 어디 있는지 모르고, 매칭도 엉뚱한 기사에게 간다. 이 실시간성을 유지하기 위해 Redis GEO(빠른 위치 저장), Kafka(폭주 흡수), WebSocket(서버 푸시)이 각자의 역할을 하고 있다는 걸 이번에 처음 흐름으로 이해했다.

단순히 "자주내면 정확하다"고 생각했는데, 기사 10,000명이 2초마다 위치를 올리면 초당 5,000건의 쓰기가 발생한다는 걸 계산해보니 왜 상태별로 주기를 다르게 해야 하는지 바로 이해됐다. 실시간성과 서버 비용은 항상 trade-off라는 것을 숫자로 느낀 순간이었다.

**둘째, 실시간성을 지키려면 상태 관리가 전제되어야 한다.**

위치가 아무리 빠르게 전달돼도, 기사와 승객이 "지금 이 호출이 어떤 상태인지"를 동일하게 알고 있지 않으면 의미가 없다. 기사가 수락했는데 서버가 모르면 두 기사가 동시에 달려가고, 승객이 취소했는데 기사가 모르면 헛걸음을 한다. 결국 실시간 위치 추적이 제대로 동작하려면 호출 상태가 항상 일관되게 유지되어야 한다는 것을 이번 설계에서 배웠다.

앞으로 시스템을 설계할 때 "이 데이터가 얼마나 빠르게 전달되어야 하는가"와 "이 상태가 어디서나 동일하게 보여야 하는가"를 먼저 물어보는 습관을 들이고 싶다.
