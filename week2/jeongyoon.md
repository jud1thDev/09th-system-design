**1. 문제 이해 및 설계 범위 확정**

**시나리오**

승객이 호출하면 주변 빈 택시를 찾아 매칭하고, 매칭된 택시가 픽업 지점에 도착할 때까지 승객은 택시 위치와 도착 예상 시간을 실시간으로 확인한다. 픽업 이후 목적지에 도착할 때까지 위치 추적이 이어진다.

해당 택시는 콘서트장, 페스티벌 행사장 앞에서 승객을 태우는 전용 택시다 ! 필자가 어렸을 적, 콘서트장에서 택시를 잡느라 호구 비용(ㅠㅠ)을 자주 지출했기 때문에 해당 택시는 매우 공정하게 운영되는 콘서트장 - 고속터미널역을 주로 오가는 택시 시스템을 가정한다. 콘서트장이라는 특정 설계가 추가되었기 때문에 해당 택시들을 당연하게도 사람들이 한번에 대규모의 사람들이 몰리게 된다.

우리는 이 택시 서비스의 이름을 이제부터 **‘EncoreTaxi’(앙코르 택시)**라고 부른다. 이 택시의 중요시되는 철학은 총 3가지. 

```tsx
- 공정한 요금
- 빠른 회전율
- 대규모 동시 호출 안정성
```

**설계 범위 (In / Out of Scope)**

| **포함 (In Scope)** | **제외 (Out of Scope)** |
| --- | --- |
| 호출 시점부터 도착 완료 시점까지의 실시간 흐름 | 회원가입 · 인증 · 기사 등록 절차 |
| 기사·승객 위치 추적 | 이용 이력 |
| 매칭 로직 | 운행 종료 후 요금 산정 · 결제 · 리뷰 · 정산 |
| 이상 상황 처리 (앱 종료, 네트워크 단절, 취소 등) | 도로망 데이터 / 경로 탐색 알고리즘 |

**시스템 구성 전제**

- 우리가 설계할 오로지 택시 호출 서비스(OOO택시)이다.
- 외부 의존: 지도 시스템(OOO맵, 경로·ETA), 회원 시스템, 결제 시스템
- 기사·승객 모두 로그인 상태, 결제 수단 등록 완료 상태로 가정
- 외부 시스템 의존에서 오는 문제는 본 서비스가 책임진다

**기능 요구사항**

- 기사 위치를 서버에 저장하고, 승객 호출 시 반경 내 빈 택시를 가까운 순으로 매칭
- 매칭 후 픽업 이동 중: 승객 화면에 택시 위치 + 픽업 지점까지 경로·ETA 실시간 표시
- 운행 중: 승객 화면에 현재 위치 + 목적지까지 경로·ETA 실시간 표시
- 이상 상황(앱 종료·네트워크 단절·호출 취소)에서 시스템이 일관된 상태 유지

**설계 대상 컴포넌트 사이의 우선순위 정하기 / 아키텍처 다이어그램 (필수)**

→ 앙코르택시는 일반 택시 서비스와는 다르게 특정 시간 / 특정 장소에서 초대형 burst trafiic이 발생 따라서 다음과 같은 컴포넌트 우선 순위를 설정

| **우선순위** | **컴포넌트** | **이유** |
| --- | --- | --- |
| 1 | Matching Service | 수만 건 동시 호출 상황에서도 빠른 배차 필요 |
| 2 | Location Service | 기사 위치를 실시간으로 안정적으로 수집해야 함 |
| 3 | Redis GEO | 반경 기반 기사 검색 성능 핵심 |
| 4 | Queue / Kafka | 호출 폭주 시 시스템 완충 역할 |
| 5 | WebSocket Server | 실시간 위치 전달 |
| 6 | External Map API | ETA 및 경로 계산 |

> 아래 8가지 질문은 답변 **선택**이지만, 본인이 짚고 넘어가는 부분을 1~2개 골라 깊이 있게 다루는 것을 권장.

**3-1. 기사 위치 업데이트 주기**

- 상태별(대기 중 · 매칭 중 · 픽업 이동 중 · 운행 중)로 어떻게 다르게 둘 것인가?

---

앙코르택시는 콘서트 종료 직후 특정 지역에 수천 대 차량이 동시에 몰리는 구조이므로 위치 업데이트 트래픽 제어가 중요함!! 그래서 기사 상태별로 위치 업데이트 주기를 다르게 설정

| 상태 | 업데이트 주기 | 이유 |
| --- | --- | --- |
| 대기 중 | 10~15초 | 실시간성보다 트래픽 절감 우선 |
| 매칭 제안 수신 | 3~5초 | 정확한 ETA 계산 필요 |
| 픽업 이동 중 | 1~3초 | 승객 UX 중요 |
| 운행 중 | 2~5초 | 실시간 경로 추적 필요 |

⇒ 대기 상태 기사가지 모두 1초 단위로 업데이트하게 되면 Redis 및 네트워크 부하가 급증할 것이다. 반면 승객이 이미 매칭된 이후에는 기사가 어디에 있는지, 실시간 이동 UX와 같은 것들이 중요해지기 때문에 더 짧은 주기를 사용

**3-2. 기사 위치 저장 — 어디에, 어떤 형태로?**

- "반경 N km 내 빈 택시를 가까운 순으로" 질의를 효율적으로 처리할 자료구조 / 저장소
- 후보: Redis GEO / Geohash / Quadtree / R-tree / PostGIS …
- 쓰기 부하 vs 읽기 부하 trade-off

---

앙코르택시는 아래의 세 가지가 핵심. 그래서 Redis GEO 기반 구조를 선택

```sql
- 반경 내 기사 검색
- 거리 기반 정렬
- 초당 대량 위치 업데이트
```

⇒ Redis GEO 선택 이유

- Redis GEO는 내부적으로 Geohash + Sorted Set 기반으로 동작하며 반경 검색, 거리 계산, 근접 정렬을 빠르게 수행할 수 있음
- 앙코르택시는 읽기 빈도가 높고 위치 데이터 휘발성이 강함 + 매우 빠른 응답이 필요하기 때문에 메모리 기반 저장소가 적합하다고 생각! → 특히 평균 2초 이내 호출 응답 + 대형 공연 종료 후 동시 검색 폭증 상황에서 디스크 기반 DB보다 적합
- 택시는 기사 위치가 지속적으로 갱신되는 구조이기 때문에 위치 저장 시스템은 높은 쓰기 처리량을 감당해야 됨 당연하게도 승객 호출 시에는 반경 Nkm 내 빈 택시 검색~ 같은 작업도 해야돼서 읽기 성능 또한 중요 → 그래서 Redis GEO을 사용!
    - 초당 대량 위치 업데이트 빠르게 처리 / GEOSEARCH 기반 반경 검색 또한 매우 빠르게 수행 가능

**3-5. 매칭 대상 선정 전략**

- 순차 제안 vs 동시 제안 (동시 수락 시 처리?)
- "가까운"의 기준 (직선거리? 도로 거리? ETA?)

---

앙코르 택시는 빠른 회전율이 중요하기 때문에 무조건 가장 가까운 기사 1명보다 빠르게 수락 가능한 기사를 확보하는 게 중요함

→ 따라서 완전 순차 방식 대신 소규모 병렬 제안 방식을 사용

```java
1. Redis GEO 기반 주변 기사 검색
2. ETA 기준 상위 N명 선정
3. 동시에 배차 요청 전송
4. 가장 먼저 수락한 기사 확정
5. 나머지 요청 자동 취소
```

⇒ 여러 기사가 동시에 수락할 수 있으므로 Redis Distributed Lock 사용

**3-8. 수요 폭주 예측 시 대응 *(예: 잠실 콘서트 종료 30분 내 2만 건 호출)***

- 사전 워밍업 / 자동 스케일링 / 가용 기사 사전 배치
- 정책 · 운영 차원 (예: 인센티브, 호출 우선순위, 분산 호출)

---

1. 호출 요청은 즉시 매칭 서비스로 전달하지 않고 다음과 같은 구조로 처리해서 순간 폭주 시에도 시스템 전체 장애를 방지
    
    ```java
    Ride Service
    → Kafka Queue
    → Matching Worker
    ```
    

2. Auto Scaling : 다음 기준 기반으로 Worker를 자동 확장

| 지표 | 의미 |
| --- | --- |
| Queue Length | 대기 호출 수 |
| Matching Latency | 평균 매칭 지연 |
| CPU Usage | Worker 부하 |
| Active Matching Count | 동시 매칭 수 |

→ 앙코르 택시는 공연 종료 시간이 사전에 예측이 가능하기 때문에 큰 행사가 있을 경우 worker를 미리 확장하고 Redis connection pool을 증가시키고 websocker 서버를 미리 scale-out 할수있음 !

**4. 설계 장점**

- Redis GEO 기반으로 빠른 반경 검색이 가능하다
    - 반경 기반 기사 탐색 / 거리 계산 / 근접 정렬을 메모리 기반으로 빠르게 수행 가능
    - 호출 요청 → 기사 검색 구간을 매우 짧게 설계 가능 / 대형 공연 종료 직후의 순간적인 검색 폭주에도 대응이 가능
- Queue 기반 구조를 통한 burst traffic 대응
    - 특정 시간, 특정 장소에 호출이 폭발적으로 몰린다는 게 앙코르택시의 가장 큰 특징인데
    - kafka / queue를 통해 호출 요청을 완충하고 비동기 처리, 워커 부하 분산이 가능, graceful degradation
- 기사 상태별로 위치 업데이트 주기를 다르게 두어서 실시간성, 네트워크 비용, Redis write 부하 사이의 균형을 맞춘다
    - 특히 대기 상태에서는 트래픽이 절감되고 픽업/운행 중에는 UX를 강화함

**5. 설계 단점**

- Redis 메모리 비용 이슈 ㅎ…
    - 메모리 기반 저장소다 보니 기사 수 증가, 위치 업데이트 빈도 증가에 따라 메모리 사용량이 빠르게 증가할 수 있음
    - 운영 비용 부담 이슈가 있을 듯 하다
- ETA 기반 매칭은 외부 지도 시스템 의존도가 높아 지도 API 지연 및 장애 발생 시 매칭 품질이 저하 될 수 있음
- 소규모 병렬 배차 방식은 빠른 매칭이 가능하고 수락률이 증가되지만 동시에 여러 기사가 수락하는 race condition 문제가 발생할 수 있음 → 필수적으로 동시성 제어가 필요
- 아무래도 콘서트 / 페스티벌 / 행사 종료 시간에 트래픽이 과도하게 집중되는 서비스다 보니 예상값보다 사용량이 떨어지는 날도 있을 것이고… 평균 트래픽이라는 게 예상하기 어려울 것
    - 또한 짧은 시간 동안 매우 높은 peak traffic을 감당해야돼서 서버 비용 증가, autoscaling하기 굉장히 복잡할 것으로 예상 됨
