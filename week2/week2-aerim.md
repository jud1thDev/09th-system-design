# Week 2 과제: 택시 호출 서비스 설계

## 1. 문제 이해 및 설계 범위 확정

### 시나리오

승객이 호출하면 주변 빈 택시를 찾아 매칭하고, 매칭된 택시가 픽업 지점에 도착할 때까지 승객은 택시 위치와 도착 예상 시간을 실시간으로 확인한다. 픽업 이후 목적지에 도착할 때까지 위치 추적이 이어진다.

### 설계 범위 (In / Out of Scope)

| 포함 (In Scope) | 제외 (Out of Scope) |
|---|---|
| 호출 시점부터 도착 완료 시점까지의 실시간 흐름 | 회원가입 · 인증 · 기사 등록 절차 |
| 기사·승객 위치 추적 | 이용 이력 |
| 매칭 로직 | 운행 종료 후 요금 산정 · 결제 · 리뷰 · 정산 |
| 이상 상황 처리 (앱 종료, 네트워크 단절, 취소 등) | 도로망 데이터 / 경로 탐색 알고리즘 |

### 시스템 구성 전제

- 우리가 설계할 오로지 택시 호출 서비스(OOO택시)이다.
- 외부 의존: 지도 시스템(OOO맵, 경로·ETA), 회원 시스템, 결제 시스템
- 기사·승객 모두 로그인 상태, 결제 수단 등록 완료 상태로 가정
- 외부 시스템 의존에서 오는 문제는 본 서비스가 책임진다

### 기능 요구사항

- 기사 위치를 서버에 저장하고, 승객 호출 시 반경 내 빈 택시를 가까운 순으로 매칭
- 매칭 후 픽업 이동 중: 승객 화면에 택시 위치 + 픽업 지점까지 경로·ETA 실시간 표시
- 운행 중: 승객 화면에 현재 위치 + 목적지까지 경로·ETA 실시간 표시
- 이상 상황(앱 종료·네트워크 단절·호출 취소)에서 시스템이 일관된 상태 유지

### 비기능 요구사항 (시간 / 지연 목표)

| 항목 | 목표 |
|---|---|
| 호출 접수 응답 시간 | 호출 요청 → "기사 검색 중" 진입까지 **2초 이내** |
| 매칭 완료 시간 | 평균 **30초 이내**, 5분 초과 시 실패 처리 |
| 위치 추적 갱신 지연 | 기사 위치 변화 → 승객 화면 반영 평균 **5초 이내** |

## 3. 상세 설계

### 우선순위 — 무엇을 깊이 팔 것인가?

본 시스템의 핵심 기술적 도전 3가지에 집중한다.

1. **호출 상태의 일관성 관리** → 배차 서비스를 단일 진실 공급원으로
2. **위치 데이터 쓰기 폭증 + 공간 검색** → Redis GEO 기반 in-memory 공간 인덱스
3. **5초 이내 실시간 반영** → WebSocket + Push 서비스 + sticky 라우팅

### 상세 아키텍처 다이어그램

<img width="2671" height="2762" alt="상세 아키텍처" src="https://github.com/user-attachments/assets/362e3eee-224e-43dd-935c-0a84ff5d9b5c" />

**데이터 흐름 (호출 ~ 매칭)**

1. 승객이 호출 버튼을 누르면
2. Load Balancer가 요청을 받고
3. API Gateway에서 인증 후 매칭 서비스로 라우팅
4. 매칭 서비스는 먼저 배차 서비스에 `REQUESTED` 상태의 배차를 생성
5. 위치 서비스를 통해 반경 내 빈 택시를 Redis GEO에서 조회
6. 후보가 정해지면 Push 서비스에 콜카드를 의뢰
7. Kafka를 통해 비동기로 기사 앱에 전달

### 컴포넌트 책임

| 컴포넌트 | 책임 | 저장소 |
|---|---|---|
| API Gateway | 진입점, 인증, 라우팅, rate limit | 무상태 |
| 매칭 서비스 | 후보 검색, 콜카드 발송, 수락 처리 | 무상태 |
| 배차 서비스 | 호출의 상태 머신, **단일 진실 공급원** | 배차 DB (PostgreSQL) |
| 위치 서비스 | 기사 위치 수신·저장·반경 조회 | 기사 위치 DB (Redis GEO), Kafka publish |
| Push 서비스 | WebSocket 연결 관리, 실시간 알림 | Kafka subscribe |

**핵심 설계 결정**: 한 호출의 상태 변화는 모두 배차 서비스를 거치게 한다. 다른 서비스가 각자 상태를 들고 있으면 일관성이 깨질 수 있으므로, "이 호출이 지금 어떤 상태야?"에 답할 수 있는 곳이 시스템에 단 하나만 존재하도록 한다.

### 상태 머신

```
[REQUESTED]
    ↓
[MATCHING]  ──→ [FAILED_NO_DRIVER]   (5분 초과)
    ↓
[MATCHED]   ──→ [CANCELED]           (승객 취소)
    ↓
[EN_ROUTE_TO_PICKUP]
    ↓
[IN_TRIP]
    ↓
[COMPLETED]
```

모든 상태 전이는 배차 서비스에 의해서만 발생하며, 다른 서비스는 배차 서비스에 상태 변경을 요청한다.

### 우선순위 1 — 호출 상태의 일관성

매칭·픽업·운행이 여러 서비스를 거치는 동안 상태가 깨지지 않도록, 모든 상태 전이는 배차 서비스를 통해서만 발생한다. 다른 서비스는 직접 상태를 바꿀 수 없다.

이상 상황 처리:

- 기사 위치 보고 30초 이상 끊김 → 매칭 중이면 재매칭
- 승객 호출 후 일정 시간 미응답 → 자동 `CANCELED`
- 매칭 5분 초과 → `FAILED_NO_DRIVER`

### 우선순위 2 — 위치 데이터 처리

데이터 특성에 맞는 저장소 분리:

- **Redis GEO**: in-memory + geohash 공간 인덱스로 `GEOSEARCH` 즉시 처리
- **영속성 보완**: AOF + 마스터-슬레이브 복제. 장애 시 기사들이 4초마다 재보고하므로 자동 복구
- **지역 샤딩**: 도시 권역별로 Redis 노드 분리

### 우선순위 3 — 5초 이내 실시간 반영

- **WebSocket**: 연결 유지로 양방향 통신, 변화 즉시 푸시
- **Push 서비스 분리**: WebSocket 연결은 인스턴스 메모리에 유지되는 상태라 다른 무상태 서비스와 섞이면 안 됨
- **Sticky 라우팅**: 같은 클라이언트의 모든 요청을 같은 인스턴스로
- **Redis 연결 매핑**: 클라이언트-인스턴스 매핑 정보 저장으로 fallback 경로 마련
- **Kafka 버퍼**: Push 서비스 장애 시 메시지가 큐에 쌓여 회복 후 처리

### 비기능 요구사항 → 설계 매핑

| 요구사항 | 달성 방법 |
|---|---|
| 호출 접수 2초 이내 | 호출 접수는 동기 응답, 매칭은 비동기 처리. 호출 수신 즉시 `REQUESTED` 상태로 배차 생성 후 클라이언트에 즉시 응답 |
| 매칭 평균 30초 / 5분 타임아웃 | 배차 서비스가 타이머 보유, 5분 초과 시 `FAILED_NO_DRIVER`로 전이 |
| 위치 반영 5초 이내 | WebSocket 기반 서버 푸시. 기사가 4초마다 위치를 보고하고, 갱신 발생 시 즉시 승객 앱에 푸시 |
| 이상 상황 상태 일관성 | 모든 상태 전이는 배차 서비스를 통해서만 발생 |

### 선택 답변 (3-1, 3-3, 3-5)

#### 3-1. 기사 위치 업데이트 주기

상태별로 위치 보고 주기를 다르게 둔다.

| 상태 | 보고 주기 | 이유 |
|---|---|---|
| 대기 중 / 매칭 중 | 매 4초 | 위치 정확도와 트래픽 균형 |
| 픽업 이동 중 / 운행 중 | 매 2초 | 정확도 우선 (승객이 실시간 추적) |
| 장시간 미운행 | 30초~1분 | 배터리·트래픽 절감 |

#### 3-3. 검색 반경과 결과 처리

- **1차 조회**: 반경 1km
- **결과 0건 시 점진적 확장**: 2km → 3km → 5km
- **5km 내에서도 0건**: 30초 대기 후 재시도
- **5분 동안 매칭 실패**: `FAILED_NO_DRIVER`로 종결

#### 3-5. 매칭 대상 선정 전략

- **동시 제안**: 후보 N명에게 콜카드 동시 발송
- **선착순 1명 수락**: 가장 먼저 수락한 기사로 확정
- **"가까운"의 기준**: ETA (직선거리 아닌 예상 소요 시간 기반)

## 4. 설계 장점

### 1. 책임 분리가 명확하다

매칭(결정)·배차(상태)·위치(추적)·Push(알림) 4개 서비스가 각자 하나의 일만 한다. 한 서비스를 고쳐도 다른 서비스에 영향이 적고, 장애가 나도 격리된다.

### 2. 호출 상태의 단일 진실 공급원

한 호출의 상태 변화를 배차 서비스가 독점 관리한다. 매칭·위치·결제 서비스가 각자 상태를 들고 있을 때 발생하는 일관성 깨짐을 구조적으로 차단한다.

### 3. 쓰기 부하와 일관성 요구의 분리

위치 데이터(초당 수천 건 쓰기, 일관성보다 신선도)는 Redis GEO에, 배차 이력(영속성·트랜잭션 필요)은 PostgreSQL에 둔다. 데이터 특성에 맞는 저장소를 골라 비용과 성능을 모두 잡았다.

### 4. Kafka로 서비스 간 결합도 낮춤

위치 서비스(publish)와 Push 서비스(subscribe)가 직접 호출 관계가 아니라 큐를 통해 비동기로 연결된다. Push 서비스가 잠시 죽어도 위치 수신은 멈추지 않고, Push가 회복되면 큐에서 밀린 메시지를 처리한다.

### 5. 무상태 서비스의 수평 확장 용이

매칭 서비스를 비롯해 API Gateway, Push 서비스 등은 모두 무상태로 설계되어 인스턴스를 자유롭게 늘릴 수 있다. 피크 시간 호출 폭주(평시의 5배)에 대응할 때 가장 부담이 큰 매칭 영역을 가장 쉽게 확장할 수 있다.

## 5. 설계 단점

### 1. 컴포넌트 수가 많아 운영 복잡도 증가

4개 서비스 + 3개 저장소 + Kafka 운영 부담이 크다. 호출 한 건이 여러 서비스를 거치므로 분산 추적(distributed tracing)이 없으면 디버깅이 어렵다. 작은 팀이라면 오버엔지니어링이 될 수 있다.

### 2. 배차 서비스가 시스템 전체의 병목·SPOF가 될 위험

모든 상태 전이가 배차 서비스를 통과한다. 배차 서비스가 느려지면 시스템 전체 처리량이 떨어지고, 죽으면 신규 호출이 막힌다.

> **보완**: 읽기 복제본을 두고 상태 조회는 복제본에서, 쓰기만 마스터로 처리. 매칭 서비스에 짧은 캐시를 두어 배차 서비스 부하를 줄임.

### 3. 외부 의존 장애가 핵심 흐름을 막을 수 있음

지도 API가 죽으면 ETA 계산이 안 되고, 회원 시스템이 죽으면 인증이 안 된다.

> **보완**: 모든 외부 호출에 타임아웃·서킷브레이커·재시도 적용. 지도 API는 결과를 짧게 캐싱하고 장애 시 직선거리 fallback. 회원 시스템은 토큰 캐시로 짧은 시간 버팀.

### 4. 위치 데이터의 일관성 보장 어려움

Redis GEO는 in-memory라 장애 시 데이터 손실 가능성이 있다. 또한 위치 서비스가 Redis와 Kafka 두 곳에 쓰는데, 한쪽만 성공하는 부분 실패(partial failure)를 다루려면 outbox 패턴 같은 추가 설계가 필요하다.

> **보완**: AOF + 마스터-슬레이브 복제로 영속성 보완. 기사가 4초마다 위치를 재보고하므로 짧은 공백은 자동 복구.

## 6. 마무리

### 6-1. 설계 요약

본 설계의 핵심은 **"한 호출의 상태는 단 한 곳에서만 관리한다"**는 원칙이다. 배차 서비스를 단일 진실 공급원으로 두고, 데이터 성격에 맞게 저장소를 분리(PostgreSQL + Redis GEO), 실시간 전달은 WebSocket 기반 Push 서비스로 처리, Kafka로 서비스 간 결합도를 낮췄다.

### 6-2. 오류가 발생하면?

| 장애 지점 | 대응 |
|---|---|
| 배차 서비스 다운 | Master-Replica 페일오버, 가장 치명적이라 다중화 필수 |
| 매칭 서비스 다운 | 무상태라 즉시 다른 인스턴스로 라우팅 |
| Redis 노드 다운 | AOF + 복제로 빠른 복구. 기사들이 4초마다 재보고하므로 짧은 공백은 자동 복구 |
| Push 서비스 다운 | Kafka에 메시지 쌓임, 회복 후 처리. 클라이언트는 자동 재연결 |
| 기사 앱 네트워크 단절 | 30초 이상 보고 없으면 연결 끊김 판단, 재매칭 |
| 외부 지도 API 장애 | 직선거리 × 평균 속도 = fallback ETA |
| 외부 결제 시스템 장애 | 운행 종료는 막지 않고 비동기 큐로 적재 |

### 6-3. 미래 규모 확장

**10배 성장 시** (일 500만 호출, 동시 운행 10만 명)

- 매칭/Push 인스턴스 증설 (무상태라 수평 확장 용이)
- Redis GEO 지역 샤딩 강화
- Kafka 파티션 증가

**지리적 확장: 단일 도시 → 글로벌**

- 도시·권역별 Redis 클러스터 분리
- 리전별 데이터센터, 트립은 한 리전에서 완결
- 사용자 인증·결제는 글로벌, 트립 데이터는 리전 로컬

### 6-4. 추가 학습

클로드에게 매칭 서비스·배차 서비스·위치 서비스를 어떤 인프라로 구성하면 좋을지 추천을 받아 정리했는데, 각 인프라를 **왜** 써야 하는지 함께 학습했습니다.

#### 1) PostgreSQL (배차 DB)

배차 데이터는 영원히 정확하게 남아야 하는 데이터다. 운행 이력은 정산·감사를 위해 영구 보존되어야 하고, 상태 전이(`MATCHING → MATCHED` 등)는 여러 필드가 원자적으로 갱신되어야 한다. 디스크 기반의 ACID 트랜잭션과 SQL 쿼리 표현력이 강한 PostgreSQL이 적합하다고 판단했다. Oracle DB도 후보였지만 라이선스 비용 부담이 크고, 이번 시스템 규모에서는 PostgreSQL로 충분하다고 보았다.

#### 2) Redis GEO (기사 위치 DB)

기사 위치 데이터는 초당 수천 건의 쓰기가 발생하고, "반경 N km 내 빈 택시"라는 공간 쿼리가 매칭의 핵심이다. 디스크 기반 DB로는 이 처리량과 응답 시간을 동시에 만족시키기 어렵다. Redis GEO는 in-memory에 geohash 기반 공간 인덱스를 내장하고 있어 `GEOSEARCH` 명령 하나로 반경 검색이 즉시 처리된다. 영속성이 약하지만 기사가 4초마다 위치를 재보고하므로 장애 후 빠르게 복구된다.

#### 3) Kafka (이벤트 큐)

서비스 간 직접 호출 대신 Kafka를 둔 이유는 **결합도 분리와 백프레셔 대응**이다. 위치 서비스가 Push 서비스를 직접 호출하면 Push 서비스 장애가 위치 수신에 영향을 준다. Kafka를 두면 위치 서비스는 publish만 하고 Push 서비스는 자기 속도로 consume할 수 있다. 또한 Push 외에 분석·모니터링 같은 다른 컨슈머도 같은 스트림을 구독할 수 있다.

#### 4) WebSocket과 Push 서비스

비기능 요구사항 "위치 변화 → 승객 화면 반영 5초 이내"를 만족시키려면 폴링 방식으로는 한계가 있었다. 폴링은 변화가 없어도 계속 요청해야 해서 트래픽 낭비가 크고, 동시 호출 수천 건이면 초당 수천 요청이 발생한다.

**WebSocket**은 클라이언트와 서버 간 연결을 끊지 않고 유지하는 양방향 통신 프로토콜이다. 서버가 변화 발생 시 즉시 푸시할 수 있어 지연이 짧고 트래픽도 효율적이다.

**Push 서비스**는 이 WebSocket 연결을 전담 관리하는 서비스다. 동시에 수천 개의 연결이 살아 있어야 하는데, 이 연결들은 특정 인스턴스의 메모리에 유지되는 "상태"이기 때문에 다른 무상태 서비스(매칭·배차)와 섞이면 안 된다. 그래서 별도 서비스로 분리했다.

#### 5) Sticky Session 라우팅

Push 서비스가 다중 인스턴스로 운영될 때 한 가지 문제가 있다. WebSocket 연결은 인스턴스 A의 메모리에 있는데, 알림이 인스턴스 B로 라우팅되면 그 연결을 찾을 수 없다.

해결책으로 **sticky session 라우팅**을 적용한다. 같은 클라이언트의 모든 요청을 항상 같은 인스턴스로 보내는 방식이다. 부하 분산 측면에서는 손해지만, WebSocket처럼 서버가 클라이언트별 상태를 유지하는 경우 필수다. 다만 인스턴스 장애 대비로 Redis에 "어떤 클라이언트가 어떤 인스턴스에 연결되어 있는지" 매핑 정보를 함께 저장해 fallback 경로를 마련한다.

### 6-5. 개인적 의견

이번 설계에서 가장 인상 깊었던 점은 **데이터마다 성격이 다르고, 그에 맞는 저장소·통신 방식을 골라야 한다**는 것이었다. "PostgreSQL 하나로 다 하면 안 되나"라는 단순한 생각에서 출발해, 위치 데이터의 쓰기 폭증과 공간 쿼리 특성을 보며 Redis GEO를, 실시간성 요구에서 WebSocket을, 그 연결 관리를 위해 Push 서비스와 sticky 라우팅까지 — 각 선택의 근거가 모두 비기능 요구사항에서 자연스럽게 따라 나왔다는 점이 시스템 설계의 본질이라는 걸 배웠다.

아는 인프라가 없으니 클로드에게 각 인프라 설명을 많이 물어봤는데, 이번 스터디를 통해서 여러 인프라와 친해지고 장단점을 알아둬야겠다는 생각이 들었다.

대학교 졸업 후 오랜만에 시퀀스 다이어그램과 아키텍처 다이어그램을 그려보니 그동안 많이 부족한 개발자였다는 것을 느꼈습니다... ㅜㅜ
