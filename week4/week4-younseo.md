# Week 4 과제: 커머스 추천 지면 시스템 설계

- 상품 조회, 클릭, 장바구니 담기 같은 사용자 행동 이벤트가 추천 결과에 어떻게 반영되는지 설계합니다.
- 후보 상품 생성, 랭킹, 캐시, 메시지 큐, 이벤트 처리, 장애 대응 전략을 비교합니다.
- 대량 트래픽 상황에서 개인화 추천 지면을 안정적으로 제공하는 구조를 확인합니다.

---

#### ⒈ 문제 이해 및 설계 범위 확정

**시나리오**

당신은 커머스 플랫폼의 추천시스템 조직에서 일하고 있다. 메인 페이지에 신규 추천 지면을 추가하는 업무를 맡았으며, 이 지면은 사용자가 관심을 가질 가능성이 높은 상품을 그리드 형태로 노출한다. (2 X 5)

- 추천 지면의 핵심 성과 지표는 CTA(장바구니 담기)이다.
- 사용자의 조회, 클릭, 장바구니 담기 이벤트를 수집할 수 있어야 한다.
- 수집된 이벤트는 이후 추천 후보 생성과 상품 우선순위 산정에 반영되어야 한다.
- 로그인 사용자와 비로그인 사용자를 모두 고려하되, 과제에서는 사용자 식별 및 노출 전략을 다룬다.

## 설계 범위 (In / Out of Scope)

---

| 포함 (In Scope) | 제외 (Out of Scope) |
| --- | --- |
| 사용자 이벤트 수집 및 처리 흐름 | 추천 모델 학습 파이프라인 |
| 추천 컴포넌트의 배치 | 추천 알고리즘의 수식/모델 구현 |
| 로그인/비로그인 사용자 식별 전략 | 결제/주문 시스템 상세 설계 |
| 추천 후보 상품 생성 흐름 | 상품 이미지/콘텐츠 제작 |
| 추천 지면에 노출되는 상품의 우선순위 산정 | 추천 지면의 UI 디자인 |
| 2 X 5 추천 지면 응답 API | 추천 지면이 플랫폼 별로 노출되는 크기 |
| 장바구니 담기(CTA) 성과 측정 | 장바구니 상품의 상세 옵션 |


## 시스템 구성 전제

---

- 사용자는 로그인 상태와 비로그인 상태를 모두 가질 수 있다.
- 상품, 재고, 가격, 카테고리 정보는 별도 상품 시스템에서 조회할 수 있다고 가정한다.
- 추천 모델은 별도 학습 파이프라인을 통해 주기적으로 전달된다고 가정한다.
- 추천시스템은 이벤트 수집, 추천 후보 조회, 랭킹, 응답 제공, 성과 측정을 책임진다.
- 추천 지면은 메인 페이지에서 호출되며, 한 번의 요청에 최대 10개 상품을 반환한다.
- 품절, 판매 중지, 노출 불가 상품은 없다고 가정한다.

## 기능 요구사항

---

- 사용자의 상품 조회, 클릭, 장바구니 담기 이벤트를 수집할 수 있어야 한다.
- 로그인/비로그인 사용자에 대한 이벤트를 연결할 수 있어야 한다.
- 수집된 이벤트는 추천 후보 생성 또는 상품 우선순위 산정에 활용될 수 있어야 한다.
- 추천 API는 메인 페이지 추천 지면에 노출할 상품 목록을 2 X 5 형태로 반환할 수 있어야 한다.
- 추천 후보가 부족하거나 추론부의 장애 상황에서도 추천 결과를 제공할 수 있어야 한다.
- 추천 결과 노출, 클릭, 장바구니 담기 이벤트를 연결해 CTA 성과를 측정할 수 있어야 한다.

## 2. 개략적 설계안

### 핵심 관찰

이 시스템에는 완전히 다른 두 개의 트래픽 흐름이 있다.

| 흐름 | 특성 | SLA |
|---|---|---|
| 추천 요청 (읽기) | 저지연, 캐시 친화, QPS 낮음 | p95 200ms |
| 이벤트 수집 (쓰기) | 대용량, 유실 방지, QPS 10배 | p95 100ms |

→ 두 파이프라인을 독립 서비스로 분리.
한 서버에 묶으면 이벤트 폭증이 추천 응답에 영향.

---

### 핵심 흐름

**추천 요청 경로 (동기):**
```
사용자 진입 → 추천 API
  → mode:{user_id} 확인 (Discovery or Navigation)
  → 해당 모드 캐시 조회
    → hit: 즉시 반환
    → miss: ML 계산 → 캐시 저장 → 반환
    → 장애 발생: 장애 케이스별 Fallback
```

**이벤트 수집 경로 (비동기):**
```
사용자 행동 → 이벤트 수집 API → 즉시 200 응답
  → Kafka (acks=all)
    → Flink (준실시간, 1분 이내)
        → 모드 전환 판단
        → Redis L3 프로필 갱신
        → Navigation 전환 시 L1 캐시 무효화
    → Spark (배치)
        → S3 적재 → ML 학습 소스
```

---

### 아키텍처

```
        ┌──────────── 사용자 ────────────┐
        │                               │
   추천 요청                          이벤트 전송
        ▼                               ▼
   [추천 API 서버]              [이벤트 수집 API 서버]
   모드 판단 + 캐시 조회          즉시 200 응답
        │                               │ acks=all
        ▼                               ▼
   [Redis]                           [Kafka]
   mode:{uid}      ← 모드 상태          │
   rec:disc:{uid}  ← Discovery 캐시    ├→ [Flink] 준실시간
   rec:navi:{uid}  ← Navigation 캐시  │    모드 전환 판단
   candidate:{uid} ← ML 후보 풀       │    캐시 무효화 신호
   profile:{uid}   ← 유저 프로필      │    L3 프로필 갱신
   popular:overall ← 전체 인기         │
   popular:cat:{}  ← 카테고리 인기     └→ [Spark] 배치
        │                                   S3 적재
        │                                   ML 학습 소스
        ▼
   [In-memory fallback]
   서버 메모리, 외부 의존 0
   Redis + ML 전체 장애 시 동작
```

## 3. 상세 설계

### 3-1. Discovery / Navigation 모드 전환

#### 모드 판단 신호

| 신호 | 해석 |
|---|---|
| 세션 시작 직후 | Discovery (기본값) |
| 같은 카테고리 클릭 3회 이상 | Navigation 전환 |
| 상세 페이지 체류 30초 이상 | Navigation 전환 |
| 검색 후 클릭 발생 | Navigation 전환 (명확한 구매 의도) |
| A 보고 뒤로가기 B 보고 뒤로가기 (비교 행동) | Navigation 전환 |
| 장바구니 담기 완료 | Discovery 복귀 |
| 세션 종료 후 재방문 | Discovery 복귀 |

#### 모드별 캐시 전략

| | Discovery | Navigation |
|---|---|---|
| TTL | 5분 | 1분 |
| 추천 기반 | 장기 취향 (L2 후보 풀) | 현재 세션 컨텍스트 |
| 갱신 트리거 | TTL 만료 | TTL 만료 + 이벤트 기반 즉시 무효화 |

Navigation TTL을 1분으로 짧게 가져가는 이유:
탐색 중인 사용자의 의도가 빠르게 바뀌기 때문.
운동화 보다가 재킷으로 관심이 옮겨가면 1분 안에 추천이 따라가야 한다.

---

### 3-2. Fallback 전략

Fallback을 두 가지로 명확히 나눈다.
**시스템 장애 Fallback**과 **추천 품질 Fallback**은 완전히 다른 문제다.

---

#### 시스템 장애 Fallback

시스템 구성 요소가 죽었을 때의 대응이다.
장애 케이스별로 다르게 처리한다.

---

**케이스 1: ML 랭킹 서버 장애**

Redis는 살아있지만 ML이 응답하지 않는 상황.
L1 캐시가 살아있으면 hit → 정상 응답. 문제는 L1 miss일 때다.

```
L1 miss 발생
  → ML 호출 (timeout 기준: 150ms)
  → ML timeout 또는 에러
    → L2 후보 풀(candidate:{uid})에서 미리 계산된 점수 순으로 꺼냄
       ML 재랭킹 없이 6시간 전 점수 그대로 사용
    → L2도 비어있으면 → popular:cat:{category} 카테고리 인기
    → 그것도 없으면 → popular:overall 전체 인기
```

ML이 죽어도 Redis의 L2 후보 풀이 있으면 서빙 가능하다.
점수가 최신이 아닐 수 있지만 빈 화면은 나오지 않는다.

---

**케이스 2: Redis 장애**

Redis가 응답하지 않는 상황.
L2 후보 풀도, L3 프로필도, 모드 상태도 모두 Redis에 있다.
ML을 호출해도 피처를 못 가져오므로 의미가 없다.

```
Redis 접근 실패 (RedisError)
  → 서버 메모리의 인기 상품 100개에서 즉시 반환
  → 외부 호출 없음. 개인화 없음. 빈 화면 없음.
```

서버 메모리 인기 상품은 서버 시작 시 S3에서 로드.
1시간마다 갱신 시도. 실패해도 기존 데이터 유지.
S3도 죽으면 코드에 하드코딩된 기획전 상품 10개 사용.

**왜 다단계 Fallback(Redis 장애 → 다른 DB → 또 다른 API)을 안 쓰나:**
Redis가 죽을 정도의 장애에서는 다른 외부 의존도 함께 불안정할 가능성이 높다.
단계마다 timeout을 기다리다가 응답 시간만 늘어난다.
외부 의존 0인 단일 경로가 진짜 안전망이다.

---

**케이스 3: 추천 API 서버 일부 인스턴스 장애**

서버 한 대가 죽는 상황.

```
로드밸런서가 health check 실패 감지
  → 해당 인스턴스로 트래픽 차단
  → 다른 인스턴스로 자동 라우팅
  → 각 인스턴스가 서버 메모리 Fallback을 들고 있으므로
     어느 인스턴스로 가도 응답 가능
```

추천 API는 stateless 설계이므로 수평 확장과 인스턴스 교체가 자유롭다.

---

**케이스 4: Flink 장애**

이벤트 실시간 처리 서버가 죽는 상황.

```
Flink 죽음
  → 이벤트 처리 멈춤
  → Kafka에 이벤트가 계속 쌓임 (Kafka는 살아있음)
  → 모드 전환 안 됨, L3 프로필 갱신 안 됨
  → 추천 자체는 기존 캐시로 정상 서빙 (추천 경로에 Flink 없음)
  → Flink 복구되면 Kafka에 쌓인 이벤트 순서대로 이어서 처리
```

Flink 장애는 추천 품질이 저하되는 것이지, 추천이 멈추는 게 아니다.

---

**장애 케이스 요약**

| 장애 | 영향 | 대응 | 개인화 |
|---|---|---|---|
| ML 랭킹 서버 | L1 miss 시 재랭킹 불가 | L2 저장 점수 → 카테고리 인기 → 전체 인기 | 부분 유지 |
| Redis | 캐시 + 후보 풀 전부 불가 | 서버 메모리 인기 상품 즉시 반환 | 0 |
| 추천 API 일부 인스턴스 | 해당 인스턴스 불능 | 로드밸런서가 다른 인스턴스로 라우팅 | 정상 |
| Flink | 모드 전환 지연 | 추천은 정상. 품질만 저하 | 저하 |

---

#### 추천 품질 Fallback

시스템은 살아있지만 개인화 후보가 부족할 때의 대응이다.
신규 유저, ML 배치 미완료, 비로그인 이력 부족 등이 해당된다.

**목표: 항상 10개 채우기**

위에서부터 채우고, 부족한 자리만 아래 단계에서 메운다.

```
1순위: ML 개인화 후보 (candidate:{uid}, L2 Sorted Set)
       이 유저에게 계산된 상품. 있으면 씀.

2순위: 관심 카테고리 인기 (popular:cat:{category})
       이 유저가 자주 본 카테고리의 인기 상품.
       1순위로 못 채운 자리를 여기서 채움.

3순위: 전체 인기 TOP 100 (popular:overall)
       그래도 부족하면 여기서 채움.
       이미 나온 상품은 제외하고 채움.
```

신규 유저는 1순위가 0개라서 2, 3순위로 전부 채운다.
이력이 쌓일수록 1순위 비중이 늘어나는 구조다.

각 단계에서 이미 선택된 상품은 제외한다.
같은 상품이 두 번 노출되는 일 없다.

### 3-3. 이벤트 수집 + CTA 퍼널

#### 이벤트 스키마

```json
{
  "event_id": "uuid",
  "event_type": "click",
  "impression_id": "imp_xyz",
  "user_id": "uid_123",
  "device_id": "did_abc",
  "item_id": "item_456",
  "category": "운동화",
  "position": 3,
  "dwell_time": 0,
  "from_search": false,
  "timestamp": 1720000000
}
```

#### impression_id: CTA 퍼널 연결

추천 API가 응답할 때 UUID 기반 impression_id를 발급한다.
프론트엔드는 추천 지면에서 발생한 이벤트에만 impression_id를 붙여서 전송한다.

```javascript
// 추천 API 응답 받음
const { impression_id, mode, items } = await fetch('/recommendations')
localStorage.setItem('impression_id', impression_id)

// 추천 지면의 상품 클릭 시
function onRecommendationClick(item) {
  sendEvent({
    impression_id: localStorage.getItem('impression_id'), // 붙임
    item_id: item.id,
    event_type: 'click'
  })
}

// 검색이나 카테고리에서 클릭 시
function onSearchClick(item) {
  sendEvent({
    // impression_id 없음 → 추천에서 발생한 게 아님
    item_id: item.id,
    event_type: 'click'
  })
}
```

impression_id가 있는 이벤트만 추천 성과로 집계된다.
impression_id에 mode 필드를 포함하면 Discovery vs Navigation 각각의 CTA 전환율을 비교할 수 있다.

#### 이벤트 신호 해석

| 이벤트 | 신호 | 모드 영향 |
|---|---|---|
| impression | neutral | 없음 |
| click | positive | Navigation 전환 카운트 |
| 체류 30초 이상 | strong positive | Navigation 전환 트리거 |
| 검색 후 클릭 | strong positive | 즉시 Navigation 전환 |
| click 후 미담기 | mild negative | 가격/옵션 불만족 시그널 |
| add_to_cart | strong positive + Discovery 복귀 | 모드 리셋 |

#### 유실 방지 end-to-end

| 단계 | 방법 | 예상 유실률 |
|---|---|---|
| 클라이언트 → API | localStorage 버퍼 + sendBeacon (이탈 시) | ~0.05% |
| API → Kafka | acks=all | ~0.01% |
| Kafka → Flink | offset commit after success | 0% |
| Flink → Redis | 멱등 자료구조 (ZADD) | 0% |
| **합산** | | **< 0.1%** |

#### 클라이언트 배치 전송 전략

| 이벤트 | 전송 방식 | 이유 |
|---|---|---|
| impression | 5초 단위 배치 | 가장 많음. 즉시 전송 시 QPS 폭증 |
| click | 즉시 전송 | Navigation 전환 트리거. 지연 불가 |
| 체류 30초 | 즉시 전송 | 전환 트리거 |
| add_to_cart | 즉시 전송 | 핵심 KPI. 유실 불가 |
| 페이지 이탈 | sendBeacon flush | 미전송 이벤트 보장 |

### 3-4. 사용자 식별

| 상태 | 식별자 | 캐시 키 | 모드 지원 |
|---|---|---|---|
| 로그인 | user_id | rec:disc:u:{uid} / rec:navi:u:{uid} | 둘 다 |
| 비로그인 | device_id (쿠키, TTL 1년) | rec:disc:d:{did} / rec:navi:d:{did} | 둘 다 |
| 콜드 스타트 | device_id (이력 없음) | — | 전체 인기로 초기화 |
| 로그인 전환 | device_id → user_id 매핑 저장 | — | 이력 소급 연결 (배치) |

비로그인 사용자도 Navigation 모드를 지원한다.
로그인 여부와 관계없이 세션 내 클릭 패턴으로 모드를 판단하기 때문이다.

### 3-5. Redis 메모리 추정

활성 사용자 500K 기준. UUID 36바이트 가정.

**L1 캐시 (Discovery + Navigation)**
```
Discovery:  피크 300 QPS × 300s × 1.1KB = ~99MB
Navigation: 피크 300 QPS × 60s  × 1.1KB = ~20MB
합계: ~120MB
```

**L2 후보 풀 (Sorted Set, 50멤버)**
```
멤버당: 36 + 8 + 40 = ~84bytes
50멤버 × 84 = 4,200bytes → 한 entry ~4.5KB
500K × 4.5KB = ~2.25GB
```

**L3 유저 프로필 (Hash)**
```
recent_clicks (50개): 1,800bytes
session_context: 400bytes
기타: 600bytes → 한 entry ~4.5KB
500K × 4.5KB = ~2.25GB
```

**합계**

| 항목 | 크기 |
|---|---|
| L1 캐시 (피크) | ~120MB |
| L2 후보 풀 | ~2.25GB |
| L3 유저 프로필 | ~2.25GB |
| 모드 상태 | ~46MB |
| 인기 상품 (overall + 카테고리 100개) | ~10MB |
| 기타 | ~50MB |
| **이론 합계** | **~4.7GB** |
| Redis 실제 (×1.5~2) | **~7~9GB** |
| **권장 인스턴스 (여유 30%)** | **12~15GB** |

HA 구성: Sentinel.
현재 규모에서 Cluster 불필요. DAU 5배 이상 시 용도별 인스턴스 분리 검토.

---

### 3-6. 대규모 트래픽 대응

#### Navigation 모드가 트래픽에 미치는 영향

Discovery만 있을 때와 달리 Navigation 모드 도입 시 ML 호출 빈도가 올라간다.

- Discovery: TTL 5분 → 5분마다 ML 호출
- Navigation: TTL 1분 + 이벤트 기반 무효화 → 더 자주 ML 호출

대응: Navigation 모드 캐시 miss 시 랭킹 서버 직접 호출 대신
candidate:{uid}(L2) + profile:{uid}(L3)로 추천 API 서버 내 in-memory 랭킹 처리.
랭킹 서버 호출 없이 ~10ms 응답 가능. 랭킹 서버 병목 방지.

#### 피크 대응

| 시간 | 부하 | 대응 |
|---|---|---|
| 07~10시, 12~13시, 18~19시, 00~01시 | 평시 ×3 | 사전 스케일링 (예측 가능) |
| 플랫폼 이벤트 | 평시 ×10~20 | 사전 인스턴스 증설 + Navigation TTL 임시 연장 (1분→3분) |

## 4. 설계 장점

- **Discovery / Navigation 이중 모드:** 단일 TTL 전략의 한계를 극복. "살 건데 찾는 중"인 사용자와 "그냥 보는" 사용자를 다르게 처리해 CTA를 모두 끌어올린다.
- **장애 케이스별 명시적 대응:** ML 장애, Redis 장애, 서버 인스턴스 장애, Flink 장애를 각각 다르게 처리. 장애 유형에 따라 개인화 수준을 단계적으로 낮추되 빈 화면은 절대 발생하지 않는다.
- **추천 품질 Fallback 분리:** 시스템 장애와 후보 부족을 다른 문제로 처리. 신규 유저도 항상 10개를 받는다.
- **impression_id로 모드별 CTA 측정:** Discovery vs Navigation 각각의 전환율을 비교해 모드 전략 효과를 숫자로 검증할 수 있다.
- **Fallback 외부 의존 0:** Redis 전체 장애 시 서버 메모리에서 즉시 응답. 다단계 외부 Fallback보다 단순하고 신뢰할 수 있다.

---

## 5. 설계 단점 / 트레이드오프

- **모드 판단 오분류 가능성:** 같은 카테고리 3회 클릭이라는 기준이 틀릴 수 있다. 구경만 했는데 Navigation으로 잘못 전환되면 추천이 좁아진다. 임계값은 A/B 테스트로 지속 검증 필요.
- **Navigation ML 호출 증가:** TTL 1분 + 캐시 무효화로 Discovery 대비 ML 호출 많아짐. in-memory 랭킹으로 부분 흡수하지만 완전 해결은 아님.
- **Redis 장애 시 개인화 0:** 전체 사용자에게 같은 인기 상품 반환. CTA 큰 폭 하락 불가피. Sentinel HA로 장애 확률을 낮추는 것이 최선.
- **ML 장애 시 L2 점수 신선도:** 6시간 전 계산된 점수로 서빙. 트렌드 반영이 늦을 수 있음.
- **Flink 장애 시 모드 전환 지연:** 복구 후 Kafka에 쌓인 이벤트를 순서대로 처리하지만, 그 사이 Navigation 전환이 안 된 사용자는 Discovery 추천을 받게 됨.
- **device_id 불안정성:** 쿠키 삭제/앱 재설치로 비로그인 사용자 영구 식별 불가. 로그인 유도가 개인화 품질에 직결.

## 6. 마무리

### 나의 개인적인 설계

**1. 추천 모드를 두 가지로**

처음에는 캐시 TTL을 얼마로 설정할지의 문제처럼 보였다. 5분이면 신선도가 떨어지고, 1분이면 ML 호출이 많아진다. 이 트레이드오프를 단일 숫자 하나로 해결하려고 하면 항상 어느 한쪽 사용자가 손해를 본다.

근데 사용자를 들여다보면 질문 자체가 잘못됐다는 걸 알 수 있다고 생각한다. "안 살 거였는데 사게 되는 사람"과 "살 건데 적합한 걸 찾는 중인 사람"은 완전히 다른 상황에 있는 거다. 이 두 사람한테 같은 TTL의 같은 추천을 주는 게 애초에 맞지 않는다고 생각해서 모드를 나누는 설계로 갔다.

모드를 나누고 나니 비기능 요구사항의 의미도 달라졌다. "이벤트 반영 1분 이내"가 단순히 빠르게 처리하라는 말이 아니라, Navigation 모드 전환이 1분 안에 추천에 반영되어야 한다는 구체적인 요구사항이 된다. 요구사항이 왜 존재하는지 이유가 생긴다는 게 중요한 포인트인 것 같다.

**2. 장애 Fallback을 케이스별로 나누기**

"Redis 죽으면 서버 메모리에서 꺼낸다" 하나로 끝내는 설계가 처음엔 단순해 보였는데, 실제로는 ML이 죽는 경우, Redis가 죽는 경우, 서버 인스턴스가 죽는 경우, Flink가 죽는 경우가 각각 영향이 다르다고 생각한다.

ML이 죽었을 때는 Redis L2 후보 풀이 살아있으면 미리 계산된 점수로 부분 개인화가 가능하다. 이걸 Redis 장애 케이스와 똑같이 처리해서 인기 상품으로 뭉뚱그려 버리면, ML 장애 시 불필요하게 개인화를 포기하는 셈이 된다. 장애 유형마다 할 수 있는 최선이 다르다는 게 이번 설계에서 배운 것 같다.
