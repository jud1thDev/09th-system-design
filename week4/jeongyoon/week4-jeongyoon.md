# Week 4 과제: 커머스 추천 지면 시스템 설계

- 상품 조회, 클릭, 장바구니 담기 같은 사용자 행동 이벤트가 추천 결과에 어떻게 반영되는지 설계합니다.
- 후보 상품 생성, 랭킹, 캐시, 메시지 큐, 이벤트 처리, 장애 대응 전략을 비교합니다.
- 대량 트래픽 상황에서 개인화 추천 지면을 안정적으로 제공하는 구조를 확인합니다.

---

### ⒈ 문제 이해 및 설계 범위 확정

**시나리오**

당신은 커머스 플랫폼의 추천시스템 조직에서 일하고 있다. 메인 페이지에 신규 추천 지면을 추가하는 업무를 맡았으며, 이 지면은 사용자가 관심을 가질 가능성이 높은 상품을 그리드 형태로 노출한다. (2 X 5)

- 추천 지면의 핵심 성과 지표는 CTA(장바구니 담기)이다.
- 사용자의 조회, 클릭, 장바구니 담기 이벤트를 수집할 수 있어야 한다.
- 수집된 이벤트는 이후 추천 후보 생성과 상품 우선순위 산정에 반영되어야 한다.
- 로그인 사용자와 비로그인 사용자를 모두 고려하되, 과제에서는 사용자 식별 및 노출 전략을 다룬다.

## 설계 범위 (In / Out of Scope)

---

| 포함 (In Scope) | 제외 (Out of Scope) |
| --- | --- |
| 사용자 이벤트 수집 및 처리 흐름 | 추천 모델 학습 파이프라인 |
| 추천 컴포넌트의 배치 | 추천 알고리즘의 수식/모델 구현 |
| 로그인/비로그인 사용자 식별 전략 | 결제/주문 시스템 상세 설계 |
| 추천 후보 상품 생성 흐름 | 상품 이미지/콘텐츠 제작 |
| 추천 지면에 노출되는 상품의 우선순위 산정 | 추천 지면의 UI 디자인 |
| 2 X 5 추천 지면 응답 API | 추천 지면이 플랫폼 별로 노출되는 크기 |
| 장바구니 담기(CTA) 성과 측정 | 장바구니 상품의 상세 옵션 |

## 시스템 구성 전제

---

- 사용자는 로그인 상태와 비로그인 상태를 모두 가질 수 있다.
- 상품, 재고, 가격, 카테고리 정보는 별도 상품 시스템에서 조회할 수 있다고 가정한다.
- 추천 모델은 별도 학습 파이프라인을 통해 주기적으로 전달된다고 가정한다.
- 추천시스템은 이벤트 수집, 추천 후보 조회, 랭킹, 응답 제공, 성과 측정을 책임진다.
- 추천 지면은 메인 페이지에서 호출되며, 한 번의 요청에 최대 10개 상품을 반환한다.
- 품절, 판매 중지, 노출 불가 상품은 없다고 가정한다.

## 기능 요구사항

---

- 사용자의 상품 조회, 클릭, 장바구니 담기 이벤트를 수집할 수 있어야 한다.
- 로그인/비로그인 사용자에 대한 이벤트를 연결할 수 있어야 한다.
- 수집된 이벤트는 추천 후보 생성 또는 상품 우선순위 산정에 활용될 수 있어야 한다.
- 추천 API는 메인 페이지 추천 지면에 노출할 상품 목록을 2 X 5 형태로 반환할 수 있어야 한다.
- 추천 후보가 부족하거나 추론부의 장애 상황에서도 추천 결과를 제공할 수 있어야 한다.
- 추천 결과 노출, 클릭, 장바구니 담기 이벤트를 연결해 CTA 성과를 측정할 수 있어야 한다.

## 3-1. 사용자 이벤트 수집은 어떻게 할 것인가?

추천 시스템의 성과(CTA)를 측정하고 실시간으로 피처를 반영하기 위해서는 ‘노출-클릭-장바구니’의 흐름을 잇는 것이 가장 중요! 

**① Impression ID (노출 식별자) 활용**

- 추천 API가 응답을 줄 때, 해당 추천 회차에 대한 고유 ID인 impression_id를 생성하여 함께 전달
- 클라이언트는 해당 지연에서 발생하는 모든 이벤트 (클릭, 장바구니…)에 이 ID를 포함하여 전송
- 유저가 원래 사려고 검색해서 장바구니에 담은 것인지 **우리가 추천해 줘서 장바구니에 담은 건지** 구별이 가능 (정확한 성과 측정)
- 잘못 누른 클릭과 진짜 살 마음이 있는 장바구니 담기를 구별
    - Flink는 장바구니 이벤트가 완성되는 즉시 가중치를 높여 유저 A의 진짜 관심사 = 아디다스 운동화 이런 식으로 업데이트를 함

```objectivec

  "impression_id": "imp_1234",
  "items": [...]
}
```

**③ 퍼널 연결 로직 (Flink)**

- Stream Join : Flink에서 impression_id를 키로 이벤트 묶기
- Window 설정 : 노출 후 최대 30분 이내의 클릭 / 장바구니 이벤트까지만 유효한 패널로 인정
- 성과 기록 : ‘노출-클릭-장바구니’가 이어진 데이터를 data lake에 저장하여 실제 CTA 전환율 기록 및 통계

④ **클라이언트 부하 최적화**

- 초당 최대 3,000건의 트래픽을 완화하기 위해 이벤트 성격에 따라 전송 방식을 다르게 함
    - 발생 빈도가 높은 '노출(Impression)' 이벤트는 클라이언트의 LocalStorage에 버퍼링 후 5초 단위로 묶어서(Batch) 전송
    - 중요한 '클릭'과 '장바구니 담기'는 즉시 전송
- 사용자가 페이지를 이탈할 때 데이터가 유실되는 것을 막기 위해 브라우저의 navigator.sendBeacon() API를 사용하여 남은 버퍼를 전송

Graceful Degradation 수행

---

### Closed 상태 (정상)

- Redis 투트랙 캐시를 모두 활용
- 사용자 맞춤형 개인화 추천 제공
- 목표 응답 속도: **< 50ms**

---

### Open 상태 (Redis 지연 / 장애 발생)

- Redis 타임아웃 및 실패율이 임계치를 초과하면 Circuit Breaker Open
- 외부 Redis 호출을 즉시 차단하여 장애 전파 방지
- API 서버 내부의 **Local In-Memory Cache**에 미리 적재해둔 **전체 인기 상품 TOP 20** 반환
- 5분 단위로 백그라운드 갱신 수행
- 목표 응답 속도: **< 5ms**

---

### Half-Open 상태 (복구 확인 단계)

- 일정 시간 이후 일부 트래픽만 Redis로 제한적으로 전달
- Redis 정상 응답 여부를 검증
- 정상 상태 확인 시 자동으로 Closed 상태 전환
- 이후 개인화 추천 모드로 복귀
</aside>

<aside>
🔥

**대응**

- Fallback 전략 - 로컬 캐시 폴백
    - API 서버는 Redis를 포기하고 즉시 자신의 서버 메모리 (로컬 캐시)에 저장되어 있는 ‘실시간 카테고리별 Top 10 상품’ 과 같은 미리 준비된 상품들을 꺼내 사용자에게 전달
    - *데이터 갱신 - 이 로컬 캐시 데이터는 5분, 혹은 일정 기간 동안 백그라운드 스레드가 비동기로 DB나 정적 파일에서 읽어와 최신화 해두도록 함*
