# Week 3 발표 준비 Q&A

## 1. 왜 Claude Code 자체 분석이 아니라 Claude Code류 서비스를 설계했는가?

Claude Code의 실제 내부 구현은 공개되지 않은 부분이 많고, 제품 기능도 계속 바뀔 수 있다. 그래서 이번 과제에서는 특정 제품의 내부를 추측하기보다, Claude Code 같은 AI 코딩 Agent가 동작하려면 공통적으로 필요한 구조를 설계했다.

발표 답변:

> Claude Code 자체를 역공학하기보다는, AI 코딩 Agent가 파일을 읽고, 명령을 실행하고, 결과를 보고 다시 판단하기 위해 필요한 공통 서버 구조를 설계했습니다. 핵심은 LLM 모델이 아니라 tool orchestration, 작업 상태 관리, streaming, sandbox입니다.

## 2. 왜 일반 채팅 서비스보다 구조가 복잡한가?

일반 채팅은 사용자 메시지에 대해 모델 응답을 생성하면 끝나는 경우가 많다. 하지만 AI 코딩 Agent는 파일 읽기, 코드 수정, 테스트 실행처럼 실제 부작용이 있는 작업을 수행한다.

그래서 다음 요소가 추가된다.

- 작업 상태 저장
- tool 실행 이력
- sandbox 격리
- 실패 복구
- 사용자에게 진행 상황 streaming
- 위험 명령 승인

발표 답변:

> 일반 AI 채팅은 응답 생성이 중심이지만, AI 코딩 Agent는 실제 프로젝트 파일과 명령 실행을 다룹니다. 그래서 단순 LLM 호출보다 작업 실행 플랫폼에 가깝고, queue, worker, event log, sandbox 같은 컴포넌트가 필요합니다.

## 3. LLM이 직접 tool을 실행하면 안 되는가?

LLM은 확률적으로 다음 행동을 생성하는 모델이다. 따라서 잘못된 명령을 만들 수도 있고, 허용되지 않은 파일을 수정하려고 할 수도 있다.

그래서 LLM은 tool 호출 의도만 반환하고, 실제 실행은 시스템이 검증한 뒤 수행해야 한다.

발표 답변:

> LLM은 어떤 tool을 쓰고 싶은지 제안할 뿐이고, 실제 실행은 Orchestrator와 Policy Engine이 검증한 뒤 Tool Worker가 수행합니다. 이 구조가 있어야 권한 제어, 감사 로그, 실패 복구가 가능합니다.

## 4. Agent Worker를 stateful하게 만들면 더 단순하지 않은가?

단일 서버에서는 stateful worker가 단순할 수 있다. 하지만 worker가 죽거나 배포로 재시작되면 작업 상태가 사라질 수 있다.

이번 설계에서는 worker 자체는 stateless하게 두고, task 상태와 event log를 외부 저장소에 저장한다.

발표 답변:

> 사용자 입장에서는 Agent가 이전 작업을 기억하는 stateful한 경험을 줘야 합니다. 하지만 인프라 관점에서는 worker를 stateless하게 두고, 상태는 DB와 Event Log에 저장하는 편이 장애 복구와 수평 확장에 유리합니다.

## 5. 왜 SSE를 선택했는가?

AI 작업 진행 상황은 대부분 서버에서 사용자 화면으로 흘러가는 단방향 이벤트다. 예를 들어 "파일 읽는 중", "테스트 실행 중", "테스트 실패" 같은 이벤트를 보내면 된다.

SSE는 HTTP 기반이라 구현과 재연결 처리가 비교적 단순하다.

발표 답변:

> MVP에서는 사용자가 서버로 계속 입력하는 양방향 터미널보다, 서버가 진행 상황을 계속 push하는 흐름이 중요합니다. 그래서 WebSocket보다 운영이 단순한 SSE를 우선 선택했습니다. 나중에 실시간 터미널 입력이 필요하면 WebSocket을 추가할 수 있습니다.

## 6. 사용자가 중간에 브라우저를 닫으면 어떻게 되는가?

Agent 작업은 브라우저 연결과 분리되어 queue worker에서 계속 실행된다. 사용자가 다시 접속하면 task id를 기준으로 현재 상태를 조회하고, 마지막으로 받은 event id 이후 이벤트를 다시 받는다.

발표 답변:

> 작업 실행과 화면 연결을 분리했습니다. 브라우저가 끊겨도 worker는 계속 실행하고, 이벤트는 Event Log에 남습니다. 재접속하면 task id와 last event id를 기준으로 이어서 볼 수 있습니다.

## 7. Worker가 작업 중 죽으면 어떻게 복구하는가?

Worker가 task를 가져갈 때 lease를 획득하고, 실행 중 heartbeat를 갱신한다. Worker가 죽으면 heartbeat가 끊기고 lease가 만료된다. 다른 worker는 만료된 task를 가져와 마지막 성공 step 이후부터 재개한다.

발표 답변:

> 각 step과 tool call 결과를 저장해두기 때문에, worker가 죽어도 처음부터 다시 시작하지 않습니다. lease가 만료되면 다른 worker가 task를 이어받고, 이미 성공한 tool call은 idempotency key로 중복 실행을 막습니다.

## 8. 같은 파일 수정 tool이 두 번 실행되면 어떻게 막는가?

각 tool call에는 `task_id`, `step_no`, `tool_call_id`로 구성된 idempotency key를 둔다. 이미 성공한 tool call이면 다시 실행하지 않고 저장된 결과를 재사용한다.

발표 답변:

> 파일 수정이나 명령 실행은 부작용이 있기 때문에 단순 retry가 위험합니다. 그래서 tool call 단위로 idempotency key를 두고, 이미 성공한 tool은 재실행하지 않도록 합니다.

## 9. 긴 로그나 큰 파일은 LLM 컨텍스트에 어떻게 넣는가?

모든 파일과 로그를 그대로 넣으면 context window를 금방 초과한다. 그래서 필요한 파일만 읽고, 긴 로그는 에러 부분과 요약을 중심으로 전달한다.

발표 답변:

> 프로젝트 전체를 LLM에 넣는 방식은 현실적이지 않습니다. 먼저 파일 목록과 검색 결과로 관련 파일을 좁히고, 긴 로그는 핵심 에러와 요약본을 컨텍스트에 넣습니다. 원본 로그는 저장소에 남기고 필요할 때 다시 조회합니다.

## 10. 위험 명령은 어떻게 판단하는가?

처음에는 allowlist와 denylist를 함께 사용한다. 예를 들어 테스트 명령은 허용하되, 시스템 디렉터리 삭제, workspace 밖 접근, 외부 전송 명령은 차단하거나 사용자 승인을 요구한다.

발표 답변:

> 모든 명령을 AI에게 자유롭게 허용하지 않습니다. Tool Registry와 Policy Engine에서 실행 가능한 명령, 접근 가능한 경로, 네트워크 사용 여부를 검사하고 위험한 작업은 사용자 승인을 요구합니다.

## 11. Sandbox는 완전한 보안을 보장하는가?

아니다. Sandbox는 위험을 줄이는 격리 계층이지 완전한 보안 솔루션은 아니다. 그래서 파일 접근 제한, 네트워크 제한, 리소스 제한, secrets redaction, 감사 로그가 함께 필요하다.

발표 답변:

> Sandbox 하나로 모든 보안 문제가 해결된다고 보지는 않았습니다. Sandbox는 격리 계층이고, 그 위에 권한 정책, 로그 마스킹, 네트워크 제한, 사용자 승인 절차를 함께 둬야 합니다.

## 12. LLM API rate limit이 걸리면 어떻게 하는가?

LLM API 호출은 Agent Orchestrator에서 rate limit을 적용하고, queue 대기열을 통해 처리량을 조절한다. 급한 작업과 일반 작업의 우선순위를 나누거나, 간단한 요약 작업은 더 저렴한 모델로 보내는 전략도 가능하다.

발표 답변:

> LLM 호출은 가장 비싸고 외부 의존성이 큰 구간입니다. 그래서 queue로 속도를 조절하고, rate limit이 걸리면 재시도 backoff를 적용합니다. 트래픽이 몰리면 대기열 안내나 모델 다운그레이드 같은 graceful degradation을 고려할 수 있습니다.

## 13. 최종적으로 이 설계에서 가장 중요한 trade-off는 무엇인가?

가장 중요한 trade-off는 자동화 수준과 안전성 사이의 균형이다.

권한을 많이 주면 Agent가 더 많은 일을 자동으로 처리할 수 있지만 위험도 커진다. 반대로 모든 행동에 사용자 승인을 요구하면 안전하지만 사용성이 떨어진다.

발표 답변:

> AI Agent 설계의 핵심 trade-off는 자동화와 통제 사이의 균형입니다. 편하게 만들수록 위험해지고, 안전하게 만들수록 사용자가 자주 개입해야 합니다. 그래서 안전한 기본 권한과 위험 작업 승인이라는 절충안을 선택했습니다.
